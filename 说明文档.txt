学生行为分析系统后端开发详细方案
一、文档目标
本方案面向“学生行为分析与风险预警平台”的后端系统开发，目标是形成一套可以支撑：
- 学校管理员端
- 学生端
- 网页端
- 安卓端
的统一后端服务体系。
后端不仅要支持模型调用，还要支持：
- 用户登录与权限控制
- 学生基础信息管理
- 学生特征管理
- 模型预测结果存储
- 个性化报告生成
- 群体画像展示
- 风险学生预警
- 批量预测与导出
换句话说，后端不是“模型脚本接口化”这么简单，而是整个平台的数据中枢和服务中枢。
二、后端总体职责
后端需要承担以下职责：
1. 用户与权限管理
系统存在至少两类角色：
- 学校管理员
- 学生
如果做更完整，可以细分成：
- 学校级管理员
- 学院级管理员
- 学生
后端必须保证：
- 学生只能看自己的数据
- 学院管理员只能看自己学院的数据
- 学校管理员可以看全校数据
2. 数据管理
负责统一管理以下数据：
- 学生基础信息
- 学生特征数据
第 1 页
- 预测结果数据
- 学生个性化报告
- 群体画像结果
- 模型指标数据
3. 模型加载与预测
后端不训练模型，只负责：
- 加载导出的模型文件
- 对齐特征
- 类别映射
- 执行预测
- 返回结果
4. 报告与建议生成
后端需要把模型结果转化为：
- 风险解释
- 学业解释
- 群体画像解释
- 干预建议
- 个性化报告
5. 统一 API 服务
为网页端和安卓端提供统一接口。
6. 批量任务支持
后端要支持：
- 批量预测
- 风险名单导出
- 学生报告导出
- 后续批量导入
7. 日志与错误处理
后端要记录：
- 登录失败
- 数据缺失
- 模型调用失败
- 数据库错误
- 导出失败
三、推荐技术栈
推荐技术栈如下：
- Web 框架：FastAPI
- 数据库：MySQL
- ORM：SQLAlchemy
- 数据校验：Pydantic
第 2 页
第 3 页
- 模型加载：joblib
- 数据处理：pandas、numpy
- 服务启动：uvicorn
技术选型原因：
FastAPI：适合做接口型项目，开发快，自带 Swagger 文档，适合比赛系统。
MySQL：适合多表关联与权限管理，评委也更容易接受。
SQLAlchemy：比直接拼 SQL 更清晰，便于后期维护。
Pydantic：和 FastAPI 原生兼容，适合定义输入输出结构。
joblib：适合加载 sklearn 模型、预处理器和聚类模型。

import pandas as pd
import numpy as np
LONG_TAIL_COLS = [
 "■■■■", "■■■■■■", "■■■■", "■■■■", "■■■■■",
 "■■■■■■", "■■■■■", "■■■■■", "■■■■",
 "■■■■■■", "■■■■■■", "■■■", "■■■", "■■■■", "■■■■"
]
def apply_category_mapping(df: pd.DataFrame, category_mappings: dict):
 df = df.copy()
 for col, mapping in category_mappings.items():
 if col in df.columns:
 df[col] = df[col].astype(str).map(mapping).fillna(-1).astype(int)
 return df
def align_features(input_dict: dict, feature_columns: list, category_mappings: dict | None = None):
 df = pd.DataFrame([input_dict])
 for col in feature_columns:
 if col not in df.columns:
 df[col] = 0
 df = df[feature_columns].copy()
 if category_mappings:
 df = apply_category_mapping(df, category_mappings)
 for c in [x for x in LONG_TAIL_COLS if x in df.columns]:
 df[c] = np.log1p(pd.to_numeric(df[c], errors="coerce").fillna(0).clip(lower=0))
 for c in df.columns:
 df[c] = pd.to_numeric(df[c], errors="coerce")
 num_cols = df.select_dtypes(include=[np.number]).columns
 df[num_cols] = df[num_cols].replace([np.inf, -np.inf], np.nan)
 df[num_cols] = df[num_cols].fillna(0)
 return df

from sqlalchemy.orm import Session
from app.models.student import StudentBasicInfo, StudentFeature
from app.models.prediction import StudentPrediction
from app.models.report import StudentReport
def get_student_full_info(db: Session, student_id: str):
 basic = db.query(StudentBasicInfo).filter(StudentBasicInfo.student_id == student_id).first()
 feature = db.query(StudentFeature).filter(StudentFeature.student_id == student_id).first()
 pred = db.query(StudentPrediction).filter(StudentPrediction.student_id == student_id).first()
 report = db.query(StudentReport).filter(StudentReport.student_id == student_id).order_by(StudentReport.id.desc()).first()
 return {
 "basic": basic,
 "feature": feature,
 "prediction": pred,
 "report": report,
 }

十三、推荐开发顺序

十五、常见坑位与规避方式
1. 训练和预测特征不一致
必须保存 feature_columns 和 category_mappings，预测时严格复现。
2. 字符串字段导致预处理报错
类别字段必须先按训练映射转成数字。
3. 接口层直接查很多表
多表组合逻辑放到 service 或 crud 层。
4. 权限控制只在前端做
第 13 页
后端必须再次校验角色和访问范围。
5. 单学生详情接口返回过少
必须一次性返回：基本信息、预测结果、核心特征、建议、报告摘要。
6. 模型加载太慢
模型在服务启动时一次性加载，不要每个请求都重新加载。

四、每个输出文件的作用
这个你也最好告诉后端。
 
1. train_features_final.csv
作用：建模用最终特征表。
用途：训练和调试使用。
后端一般不直接依赖整表在线预测，但可以作为测试样本来源。
 
2. risk_model.joblib
作用：风险学生识别模型。
输出：
• 风险标签
• 风险概率
• 风险等级
 
3. scholarship_model.joblib
作用：奖学金获得概率预测模型。
输出：
• 是否可能获得奖学金
• 概率值
 
4. performance_model.joblib
作用：学业表现分类模型。
输出：
• 高表现 / 中表现 / 低表现
 
5. health_model.joblib
作用：健康水平分类模型。
输出：
• 高健康 / 中健康 / 低健康
 
6. change_trend_model.joblib
作用：成绩变化趋势预测模型。
输出：
• 上升 / 稳定 / 下降
 
7. score_regression_model.joblib
作用：综合成绩回归预测模型。
输出：
• 预测分数
 
8. cluster_model.joblib
作用：学生画像聚类模型。
输出：
• 聚类编号
• 学生画像类别
 
9. feature_schema.json
作用：定义每个模型输入哪些特征列。
后端必须按这个顺序和字段喂模型。
 
10. category_schema.json
作用：定义类别字段编码映射。
后端必须按这个映射把字符串转成数字，再调用模型。
 
11. cluster_name_map.json

作用：定义聚类编号与中文名称的对应关系。
后端展示时要用它。

• .joblib

• cluster_name_map.json
第三层：调用逻辑
