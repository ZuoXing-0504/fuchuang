# 知行雷达系统使用说明

## 1. 文档说明

本文档面向企业客户、项目验收人员及系统运维人员，说明“知行雷达”系统当前服务器部署版本的组成、访问方式、核心功能、日常使用流程及运维要点。

本文档对应的是已经部署在服务器上的可访问版本，不包含本地开发环境搭建、移动端打包流程或源码调试说明。

## 2. 系统简介

知行雷达是一套面向高校学生管理场景的数据分析与风险预警系统。系统基于学生行为数据、分析结果文件和模型资源，对学生群体进行画像划分、风险识别、趋势分析和报告展示，为学校管理部门提供辅助决策支持。

当前交付版本已经完成服务器部署，具备以下能力：

- 管理端在线访问与登录
- 学生端在线访问与登录
- 风险名单与重点学生筛查
- 学生画像、趋势分析与图表展示
- 学生详情页与完整报告查看
- 预测模块与手动输入再预测
- 数据库账号体系与日志记录

## 3. 当前部署架构

服务器端采用容器化部署，主要组成如下：

- 后端服务：Flask + Gunicorn
- 数据库：PostgreSQL
- 网关与静态资源服务：Nginx
- 管理端前端：Web Admin
- 学生端前端：Web Student
- 模型与资源文件：
  - `ml/saved_models`
  - `our project/analysis_master.csv`
  - `our project/train_features_final.csv`
  - `our project/model_outputs`
  - `our project/analysis`

系统通过 Nginx 对外统一提供访问入口，浏览器和移动端均通过 HTTP 接口访问后端能力。

## 6. 核心功能说明

### 6.1 管理端

管理端用于学校管理人员查看全局概览、风险名单、分析结果及学生详情，主要包含以下模块：

- 总览首页
  - 查看样本总量、风险等级分布、注册覆盖情况
  - 查看分析模块摘要与重点学生概览
- 风险名单
  - 筛选已注册与未注册学生
  - 按学院、专业、风险等级、画像类别进行检索
  - 进入学生详情页与完整报告
- 院系对比
  - 查看不同学院和专业之间的画像与风险差异
- 干预工作台
  - 辅助识别需要重点跟进的学生对象
- 预测模块
  - 基于已有特征和模型进行预测展示
- 分析成果
  - 查看全样本分析图表、关键结论及图表详情
- 设置中心
  - 查看系统基本配置与状态信息

### 6.2 学生端

学生端用于展示学生个人画像、分析结果和报告内容，主要包含以下能力：

- 学生个人概览
- 画像结果与标签说明
- 趋势分析
- 报告查看
- 预测结果查看与手动输入再预测
- 接口地址设置（移动端场景）

## 7. 推荐使用流程

### 7.1 管理员使用流程

1. 打开管理端登录页。
2. 使用管理员账号登录系统。
3. 在首页查看样本总量、风险分布和注册覆盖情况。
4. 进入风险名单，筛选重点学生。
5. 点击学生详情或完整报告，查看个体画像与解释结果。
6. 进入分析成果页，查看图表和关键结论。
7. 根据需要进入预测模块或干预工作台，进行进一步分析。

### 7.2 学生使用流程

1. 打开学生端登录页。
2. 使用学生账号登录。
3. 查看个人画像、标签、趋势和报告。
4. 如需展示预测能力，可进入预测模块修改输入项并重新预测。

## 9. 启动与检查方式

### 9.1 启动服务

在服务器项目目录执行：

```bash
cd /home/ubuntu/fuchuang
sudo docker-compose -f docker-compose.prod.yml --env-file .env.production up -d --build
```

### 9.2 旧版 Docker Compose 异常处理

若旧版 `docker-compose` 在重建容器时出现 `ContainerConfig` 相关报错，可先删除旧容器后再启动：

```bash
cd /home/ubuntu/fuchuang
sudo docker-compose -f docker-compose.prod.yml --env-file .env.production rm -sf backend web-admin web-student nginx
sudo docker-compose -f docker-compose.prod.yml --env-file .env.production up -d backend web-admin web-student nginx
```

### 9.3 查看容器状态

```bash
sudo docker-compose -f docker-compose.prod.yml --env-file .env.production ps
```

正常情况下应看到：

- `fuchuang_backend_1` 为 `Up (healthy)`
- `fuchuang_postgres_1` 为 `Up (healthy)`
- `fuchuang_nginx_1` 为 `Up`
- `fuchuang_web-admin_1` 为 `Up`
- `fuchuang_web-student_1` 为 `Up`

### 9.4 健康检查

```bash
curl http://127.0.0.1/healthz
curl http://127.0.0.1/api/health
```

若返回状态正常，即表示 Nginx、后端与数据库链路可用。

### 9.5 查看后端日志

```bash
sudo docker-compose -f docker-compose.prod.yml --env-file .env.production logs --tail=120 backend
```

## 10. 日常运维建议

### 10.1 数据或模型更新后页面未及时变化

如替换了模型文件、更新了分析数据、迁移了账号数据，但页面未立即体现，建议重启后端容器以刷新运行时缓存：

```bash
sudo docker-compose -f docker-compose.prod.yml --env-file .env.production restart backend
```

### 10.2 图表或前端页面异常

若前端页面出现资源缓存导致的错页、旧页面残留、学生端与管理端资源串用等现象，建议：

- 先重建对应前端容器
- 再使用无痕窗口访问
- 必要时同时重建 `nginx`

### 10.3 账号数据迁移

如需将历史 SQLite 账号数据迁入 PostgreSQL，可将历史数据库文件上传至服务器后执行迁移脚本。当前项目已完成一次迁移验证，账号和日志均可正常迁入。

## 12. 使用注意事项

- 服务器安全组需放行 `80` 端口。
- 若后续接入 HTTPS，需额外放行 `443` 端口。
- 建议企业正式上线前配置独立域名与 HTTPS 证书，以提升访问稳定性与安全性。
- 如移动端继续作为正式交付内容，建议统一使用域名 API 地址，而非公网 IP。
